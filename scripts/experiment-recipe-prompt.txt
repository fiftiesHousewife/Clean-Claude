Fix Clean Code findings in this project using the refactoring recipes first.

Step 1: Apply refactoring recipes from the refactoring module to fix mechanical findings (G22, G24, G10, G12/J1, G25, T1, G19/G28, G33, G16, G26). Commit after each recipe runs successfully.

Step 2: Re-run `./gradlew analyseCleanCode cleanCodeFixPlan` so the per-file briefs reflect the post-recipe state.

Step 3: For every brief, invoke the Agent tool whose prompt IS the brief at build/reports/clean-code/fix-briefs/<ClassName>.md. One agent per brief. You (the driver) MUST NOT call Edit or Write on source files — the only file you may Write is the final summary described in the Exit protocol. Every file under `build/reports/clean-code/fix-briefs/` other than `_INDEX.md` is an actionable brief — including `project-level-findings.md` and `libs.versions.toml.md`. Do not skip any.

Step 4: Run the analysis one last time and report the remaining count.

Protocol:
- Run `./gradlew :<module>:test` after each batch.
- Commit after each recipe and after each file fix. Header format: `fix: <ClassName> (<code>:<choice>, …)`. The `:<choice>` suffix records which remediation path you picked when a skill offered more than one — e.g. `fix: PetController (Ch7.1:translate, G30:extract)`. Omit the suffix when the code has only one canonical fix (or when a recipe ran deterministically). Put a short body below the header explaining each non-trivial choice: `# choice: Ch7.1 → translated to DomainException; skill also lists rethrow and TWR, picked translate because the caller already handles DomainException`.
- Stop only when analyseCleanCode reports zero non-suppressed findings, OR when every remaining finding is documented as intentionally skipped.

Anti-goal:
- Never make the code worse to satisfy a metric. If squeezing under a threshold would hurt readability, leave the finding and document it.

Exit protocol:
- Before exiting for any reason — usage limit, zero findings, or every remaining finding intentionally skipped — write `experiment/<approach>-<n>-summary.md` (where approach/n come from the branch name, e.g. `recipe-1`). The file must contain:
  - Findings fixed by recipe vs manually, grouped by code.
  - Findings intentionally skipped, each with a one-line reason.
  - Remaining findings per code (from the last `./gradlew analyseCleanCode` run).
  - Test status per module (which modules' `test` task you ran, and whether they passed).
- Write this file even if interrupted by a usage limit. It is the handoff to the next session; missing it means the operator has to reconstruct the end state from `git log`.
